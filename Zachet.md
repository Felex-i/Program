- V1-C-19:
1) Порядок действий: Поток открывается функцией `fopen(filename, "r")`. Результат вызова обязательно проверяется на равенство `NULL` (ошибка открытия файла). Чтение данных производится в цикле до наступления конца файла (`EOF`). По окончании работы поток принудительно закрывается функцией `fclose(file)` для освобождения системных ресурсов.
2) Переменные: Понадобятся переменная типа `double sum = 0.0;` для накопления суммы, целочисленный счётчик `int count = 0;` для подсчёта количества успешно считанных значений, а также буферная переменная (`int temp;`) для текущего элемента.
3) Анализ ошибок и отсутствие данных: При чтении проверяется возвращаемое значение функции (например, `fscanf`). Если оно равно `1`, это означает успешный парсинг числа — значение добавляется в `sum`, а `count` увеличивается. Если по завершении цикла значение `count == 0`, программа корректно обрабатывает исключительную ситуацию (предотвращая деление на ноль) и сообщает пользователю, что в файле отсутствуют валидные числовые данные. В противном случае вычисляется результат `sum / count`.

### Развернутые ответы
- V1-R-09:
1) Интерфейс элемента: Сначала создаётся общий интерфейс поведения на базе типа `trait`. Назовём его `Reportable` с методом `fn to_report_string(&self) -> String;`. Каждый элемент коллекции обязан реализовать этот метод.
2) Ограничение типа (Trait Bounds): Обобщённая функция объявляется с использованием generic-параметра типа `T`, на который накладывается ограничение: `fn generate_report<T: Reportable>(items: &[T])`. Это гарантирует компилятору, что к любому элементу внутри функции можно безопасно применить метод `to_report_string`.
3) Переиспользуемость: Такой подход разделяет алгоритм генерации отчёта и внутреннее устройство конкретных типов данных. Если в систему добавляется совершенно новая структура, достаточно написать для неё `impl Reportable for NewStruct`. Сам код функции `generate_report` при этом остаётся абсолютно неизменным и не требует перекомпиляции, обеспечивая строгое соблюдение принципа открытости/закрытости (Open-Closed Principle).

- V1-R-10:
1) Операции в unsafe: Блок `unsafe` открывает доступ к пяти специфическим действиям: разыменованию сырых указателей (`*const T` / `*mut T`), вызову небезопасных функций (включая внешние C-функции через FFI), реализации небезопасных трейтов (`unsafe trait`), изменению или чтению изменяемых статических переменных (`static mut`), а также доступу к полям объединений (`union`).
2) Сохраняющиеся гарантии: `unsafe` вовсе не отключает компилятор. Проверка типов, правил областей видимости, а также фундаментальный анализ заимствований (Borrow Checker) для безопасного кода внутри блока продолжают работать в штатном режиме. Контекст `unsafe` означает лишь то, что компилятор не может автоматически гарантировать безопасность памяти для пяти вышеуказанных операций, поэтому программист берёт на себя обязанность доказать отсутствие неопределённого поведения (UB).
3) Минимизация объёма: Объём `unsafe`-кода должен быть строго минимальным. Это позволяет локализовать места возможных дефектов (таких как порча памяти или race conditions). Хорошей практикой в Rust считается инкапсуляция минимального `unsafe`-кода внутри безопасных абстракций (Safe API), чтобы вызывающий прикладной код оставался полностью защищённым.
